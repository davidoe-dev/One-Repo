# The One-Repo Manifesto: Conversational Driven Development

**Problem solved.**

From today, the specification and the implementation share a single, undeniable source of truth. One repository. One continuous AI dialogue. The documentation *is* the code, and the code *is* the documentation. 

The decades-old war between business intent and executable code is over. For too long, we have accepted a broken reality: process documentation rotting in a disconnected wiki the moment a hotfix is pushed to the codebase. By merging these two worlds, we fix the foundation.

### I. One Source of Truth
Human-readable process documentation and executable code must share the exact same Git repository. We require this because wikis lie. Disconnected documents become outdated and dishonest the second new code is merged. By housing them together, we eliminate the illusions. When a pull request is merged, the business rule and the application logic advance together. What you read is exactly what the system does.

### II. Nothing is Hidden
Code is no longer typed in the dark; it is negotiated in the open. The workflow transitions from solitary programming to a transparent conversation with an AI orchestrator. You articulate the change in business logic, and the AI translates that dialogue into parallel updates for both the documentation and the code. You cannot change the system without updating the rules. Because the dialogue continuously writes both, every single architectural decision is exposed and documented.

### III. The Honest Dialogue
Modern development fails when the context window breaks. By enforcing the single-repository rule, the AI orchestrator is never blind. It ingests the "Why" (the process docs) and the "How" (the code) as a unified entity within its context window. This complete visibility is what keeps the interaction honest. If a developer attempts a quick fix that violates established business rules, the AI flags the divergence immediately. The architecture becomes self-healing because the AI has the holistic context required to hold the system accountable.

### IV. The Version-Controlled Truth
Engineers take version control for granted, but for process documentation, it is a revolution. Wikis have no true memory; they are easily overwritten and silently altered. By putting the business process into Git, every single change in strategy, logic, or compliance is tracked, time-stamped, and permanently linked to the exact code it altered. You can instantly rewind the entire system—both what it did and *why* it did it—to any precise second in history.

---

### The Era of Honest Software
This is the end of technical debt born from miscommunication. The future of engineering software does not rely on better wikis, stricter manual compliance, or endless sync meetings; it relies on undeniable proximity. By unifying business intent and executable code within a single repository, we stop translating the past and start generating the future. The truth is no longer scattered across disconnected platforms. It is in the repo. Adopt the dialogue, expose the logic, and build systems that are finally honest.
